Inventory intake API documentation

Endpoint

POST /v1/acquisition/intake/{referenceId}

Defaulting and validation

The Inventory intake API applies the same defaulting and validation logic as the Intake entry screen in the Web interface. This section outlines additional rules and behaviours specific to using the API for the POST method, while still aligned with the interface logic.

Transaction category

  • If only one valid category type exists, the input can omit this value, and the system will apply the default.
  • If multiple category types exist:
    • If a valid value is provided, it will be used.
    • If no value is provided and CFG_TXN_CATEGORY is active and has a valid value, that value will be used.
    • If no value is provided and CFG_TXN_CATEGORY is inactive, an error will be returned.

Input payload

  • If no input payload is provided, a full intake will be assumed.
  • For partial intake, the payload must include:
    • Line identifier
    • Quantity received
    • Agreed unit cost
  • Only existing lines from the purchase reference can be processed.

Delivery Date

  • If CFG_DEFAULT_DELIV_DATE is "ON", the delivery date is auto-populated unless specified.
  • If "OFF", the delivery date must be explicitly provided.
  • If missing or invalid, an error will be returned.

Over-Receipt Rules

  • For quantity-based and stock-tracked items, over-receipt is not permitted.
  • For non-quantity-based items:
    • If CFG_ALLOW_OVER_RECEIPT is active: over-receipt is blocked.
    • If inactive: over-receipt is allowed.

Inventory Tracking Fields

  • Applicable only to stock-tracked items: storageZone, locationCode, batchCode, serialIdentifier
  • For non-stock items.
    Note: These fields are ignored.
  • For stock-tracked items:
    • storageZone and locationCode are mandatory.
    • serialIdentifier must be unique per item and limited to 20 characters.
    • Serial identifiers cannot be reused within the same intake record.
  • For items with batch and expiry tracking enabled, date fields can be included in batch data.

Parameters

Name Location Description Required Type
referenceId path The purchase reference ID. Must be in a valid state. Yes long
intakePayload body The intake record to be created, formatted as JSON. Yes object
organisationId query Identifier for the organisation initiating the request. No string
transactionCategory query Category type for the transaction. Conditional string
accessToken header Authentication token. No string

Response Codes

  • 201 - Created
  • 400 - Bad request
  • 404 - Not found
  • 410 - Resource no longer available
  • 422 - Unprocessable entity
