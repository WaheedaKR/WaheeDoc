Inventory intake API documentation

Endpoint

POST/v1/acquisition/intake/{referenceId}

Defaulting and validation

The Inventory intake API applies the same defaulting and validation logic as the Intake entry screen in the (Name of product) Web interface. The following rules describe behaviour specific to using the API with the POST method, while remaining consistent with the interface logic.

Transaction category

The system determines the transaction category based on the following rules:

  • If only one valid category type exists, the system applies it automatically when the value is omitted.
  • If multiple valid category types exist:
    • If a valid category value is provided in the request, the system uses it.
    • If no value is provided and the CFG_TXN_CATEGORY parameter is active with a valid value, the system uses that configuration value.
    • If no value is provided and the CFG_TXN_CATEGORY parameter is inactive, the system returns an error.

Tip: If your organisation frequently defaults to the same category, enabling the CFG_TXN_CATEGORY parameter simplifies API integrations by reducing required input.

Input payload

The API interprets the payload according to the level of detail provided:

  • If the request contains no payload, the system assumes a full intake.
  • For a partial intake, the payload must include:
    • Line identifier
    • Quantity received
    • Agreed unit cost
  • The API processes only the lines that exist in the referenced purchase document.

Note: Partial intake requests must not introduce any new lines. All referenced lines must match exactly with the originating purchase document.

Delivery date

The delivery date is validated and defaulted according to the following configuration settings:

  • If the CFG_DEFAULT_DELIV_DATE parameter is set to ON, the delivery date is auto-populated unless provided in the request.
  • If the configuration is OFF, the delivery date must be explicitly provided.
  • If the delivery date is missing or invalid, the system returns an error.

Tip: When integrating external logistics systems, ensure all timestamps are converted to the organisation’s expected date format to avoid validation issues.

Over-receipt rules

The system applies the following rules when determining whether over‑receipt is allowed:

  • Over‑receipt is never permitted for quantity‑based or stock‑tracked items.
  • For non‑quantity‑based items:
    • If CFG_ALLOW_OVER_RECEIPT is active, over‑receipt is blocked.
    • If the configuration is inactive, over‑receipt is permitted.

Note: Over‑receipt checks are applied at line level. If multiple lines exist, each line is evaluated independently.

Inventory tracking fields

The following fields apply to stock‑tracked items only:

  • storageZone, locationCode, batchCode, serialIdentifier

Additional rules for inventory tracking fields:

  • For non‑stock items, all inventory tracking fields are ignored.
  • For stock‑tracked items:
    • storageZone and locationCode are mandatory.
    • serialIdentifier must be unique, limited to 20 characters, and cannot be reused within the same intake record.
  • For items with batch and expiry tracking enabled, relevant date fields may be included in the batch information.

Parameters

Name Location Description Required Type
referenceId path The purchase reference ID. The reference must be in a valid state. Yes long
intakePayload body The intake record to create, formatted as JSON. Yes object
organisationId query Identifier for the organisation submitting the request. No string
transactionCategory query The transaction category to apply. Conditional string
accessToken header Authentication token. No string

Response codes

  • 201 – Created
  • 400 – Bad request
  • 404 – Not found
  • 410 – Resource no longer available
  • 422 – Unprocessable entity
